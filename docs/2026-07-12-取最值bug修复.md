# Bug 修复：aggregation_type="level" 节点多条申报时错误求和

**日期**：2026-07-12  
**影响范围**：核算第二阶段（Phase 2）自底向上汇总  
**严重程度**：高 — 导致所有 `level` 类型叶子申报节点的多条申报分数被错误求和，进而使父级 `max` 节点的取最值失效

---

## 问题描述

### 场景

学生在"项目成果认定"节点下提交了 2 条申报：
- 申报 A："国家级科研训练-主持-结题"，对应分数 **1.0**
- 申报 B："国家级科研训练-主持-优秀"，对应分数 **2.0**

### 规则树结构

```
科研成果 (sum)
├── 项目成果 (max)           ← 期望：取子节点最高分
│   └── 项目成果认定 (level, is_apply_entry=1)
├── 论文成果 (max)
│   └── 论文成果认定 (level, is_apply_entry=1)
└── 其他成果 (max)
```

### 预期行为

- 项目成果认定：多条申报取最高 → **2.0**
- 项目成果：子节点传回 2.0 → **2.0**
- 科研成果：子节点求和 → **2.0**

### 实际行为（修复前）

- 项目成果认定：`aggregation_type="level"` 未被 Phase 2 识别，**默认 SUM** → 1.0 + 2.0 = **3.0**
- 项目成果：`max` 只收到一个值 3.0 → **3.0** ❌
- 科研成果：sum(3.0) → **3.0** ❌

### 根因

Phase 2 聚合循环仅处理了 `max` 和 `cap` 两种 `aggregation_type`：

```javascript
if (node.aggregation_type === "max") {
  raw = inputs.length ? Math.max(...inputs.map(toNumber)) : 0;
}
```

`level` 类型节点（叶子申报项，`is_apply_entry=1`）未匹配任何分支，走默认 SUM 路径，多条申报分数被错误累加。父级 `max` 节点只收到已求和的单个值，取最值形同虚设。

---

## 修复方案

在 Phase 2 聚合判断中，将 `level` 与 `max` 同等对待——取直接输入的最大值：

```javascript
// 修复前
if (node.aggregation_type === "max") {

// 修复后
if (node.aggregation_type === "max" || node.aggregation_type === "level") {
```

### 设计理由

- **`max`**：汇总节点，子节点可能是汇总项或申报项，取最高分
- **`level`**：叶子申报节点，每条申报已通过 Phase 1 按等级配置算出独立分数；多条申报时只计最高一条（每类成果取最优）
- 无需递归，仅处理当前节点的直接输入（`ownScores` + `childResults`），简洁高效

### 受益节点

默认规则集中所有 `level` 类型的叶子申报节点：
- `innovation.research.project.result`（项目成果认定）
- `innovation.research.paper.publication`（论文成果认定）
- `innovation.competition.professional.award`（专业竞赛获奖认定）
- `innovation.competition.creative.award`（创意类竞赛获奖认定）
- 其他所有 `aggregation_type="level"` 节点
