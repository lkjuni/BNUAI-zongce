# 人工智能学院本科生综合测评自动算分系统 - 接口设计文档

---

## 一、概述

### 1.1 系统说明
本系统仅计算综合测评的**加分部分**（学术创新成果 + 学生工作），不涉及学业成绩测算。学业成绩部分由学院另行处理。

系统采用前后端分离架构，后端提供RESTful API接口，前端为Web应用。AI辅助识别模块独立部署，通过内部API调用。综合测评加分在后台自动计算。

本学院不设班级，每个专业仅一个班，**行政班 = 年级 + 专业**（如"2023级计算机科学与技术"）。系统不单独存储班级实体，通过年级+专业组合逻辑确定行政班归属。

### 1.2 基础信息
- **Base URL**: `/api/v1`
- **认证方式**: JWT Token（Bearer Auth），登录后获取，有效期2小时
- **请求格式**: `application/json`（文件上传使用 `multipart/form-data`）
- **响应格式**: 统一响应体
- **字符编码**: UTF-8

### 1.3 统一响应体结构

> 以下示例中所有数据均为占位示意，每行末尾注释说明该字段含义。

```json
{
  "code": 200,                                    // 业务状态码
  "message": "操作成功",                            // 提示信息
  "data": {},                                     // 响应数据（类型视接口而定）
  "timestamp": 1694160000000                      // 响应时间戳（毫秒）
}
```

### 1.4 通用错误码

| 错误码 | 说明 |
|--------|------|
| 200 | 成功 |
| 400 | 请求参数错误 |
| 401 | 未认证/Token过期 |
| 403 | 权限不足 |
| 404 | 资源不存在 |
| 409 | 数据冲突（如重复提交） |
| 500 | 服务器内部错误 |

## 二、用户与认证模块

### 2.1 角色定义

| 角色 | 角色码 | 权限范围 |
|------|--------|----------|
| 学生 | STUDENT | 查看本人信息、填报加分项并上传证明材料、在公示期查看本人得分明细（含审核状态、不通过原因、公示等级） |
| 班委 | CLASS_COMMITTEE | 审核本行政班（同年级+同专业）同学材料并写明不通过原因、查看本行政班得分汇总表、发起行政班公示 |
| 学院管理员 | COLLEGE_ADMIN | 复核全院所有同学申报材料、发起学院公示、修改综测细则（条目修改和分数修改）、统一上传学院加分及证明材料（通过xlsx文件批量导入）、导出全院数据、用户管理、密码修改等 |

> **角色合并说明**："班主任""学业辅导室""辅导员"合并为同一角色"学院管理员"，拥有全院复核、规则配置、数据导出、用户管理等完整管理权限。

### 2.2 认证接口

#### 2.2.1 登录

```
POST /api/v1/auth/login
```

**请求体**:
```json
{
  "username": "<学号或工号>",                      // 学生用学号登录，教职工用工号登录
  "password": "<登录密码>"                         // 密码，首次登录后强制修改
}
```

**响应**:
```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
    "token": "<JWT_TOKEN>",                       // JWT令牌，后续请求放入Authorization头
    "expiresIn": 7200,                            // 令牌有效期（秒），2小时
    "user": {
      "userId": "<用户唯一ID>",                    // 系统内部用户ID
      "username": "<学号或工号>",                  // 登录用户名
      "name": "<用户姓名>",                        // 真实姓名
      "role": "<角色码>",                          // STUDENT / CLASS_COMMITTEE / COLLEGE_ADMIN
      "grade": 2023,                              // 年级（学生有此字段，教职工为null）
      "major": "<专业名称>"                        // 专业（学生有此字段，教职工为null）
    }
  },
  "timestamp": 1694160000000
}
```

#### 2.2.2 退出登录

```
POST /api/v1/auth/logout
```

**请求头**: `Authorization: Bearer <token>`

**响应**: 成功响应，无data负载

#### 2.2.3 获取当前用户信息

```
GET /api/v1/auth/me
```

**请求头**: `Authorization: Bearer <token>`

**响应**: data结构与登录接口中的user对象一致

#### 2.2.4 修改密码

```
PUT /api/v1/auth/password
```

**请求头**: `Authorization: Bearer <token>`

**请求体**:
```json
{
  "oldPassword": "<旧密码>",                       // 当前密码
  "newPassword": "<新密码>"                        // 新密码
}
```

```json
    "currentPublicityStage": "<当前公示阶段码>",    // 公示阶段码，见7.2
    "publicityDisplay": "<公示阶段中文描述>",       // 公示阶段中文描述
    "scores": {
      "academicInnovation": {
        "score": 0.0,                             // 学术创新得分
        "maxScore": 7.0,                          // 学术创新满分上限
        "items": [
          {
```

> **设计说明**：依据综测细则附件四，党团班活动的加减分依托评议证明材料确定。实际场景中，班委评议后出具证明文件，党支委和团支委的评议同理。
>

#### 5.1.1 上传党团班活动加减分（含证明文件）

```
POST /api/v1/academic-years/{yearId}/party-class-score
```

**请求头**: `Authorization: Bearer <token>`（仅班委或学院管理员可操作）

**权限说明**: 班委只能上传本人管辖行政班（同年级+同专业）的数据；学院管理员可上传全院任意行政班的数据。学生角色无此接口权限。

**请求体** (`multipart/form-data`):

- `file`: 评议证明文件（支持 pdf/jpg/png，最大10MB。如班级活动次数统计证明、党支部活动出勤证明等，文件中已有评议单位签字确认）
- `description`: 加减分描述（字符串，简洁说明本次加减分结论。如"全班39人均满足班级活动出勤2/3标准，每人+0.5分"、"以下3名党员出勤不足1/2，各-0.5分"）
- `records`: 加减分明细（JSON字符串）

**records JSON结构**:
```json
[
  {
    "studentId": "<学号>",                         // 学生学号
    "scoreChange": 0.0                             // 分数变化（正数为加分，负数为扣分，0为不变化）
  }
]
```

> **示例场景**：2023级计算机师范班全班39人，班委出具证明判定均满足出勤2/3标准。
> - `file`: 上传 `2023级计算机师范班班级活动次数统计证明.pdf`
> - `description`: "全班40人均满足班级活动出勤2/3标准，每人+0.5分"
> - `records`: 包含39个对象的数组，每个对象 `scoreChange` 为 `0.5`

```json
{

  }

```

## 六、智能核算引擎模块（后台自动计算）

> **设计说明**：综合测评加分由系统在后台自动计算。当学生提交申报材料、班委审核通过后，系统自动触发重新计算。核算结果无需手动发起，学生和班委可直接查看。本系统仅核算加分部分，不含学业成绩。

### 6.1 自动核算触发机制

核算在以下时机自动触发：
1. 学生提交新申报或修改申报后
2. 班委审核通过/退回申报后
3. 学院管理员审核通过/退回申报后
4. 学院管理员修改规则条目或分数后
5. 学院管理员批量上传学院加分后

## 十六、接口设计原则与说明

### 16.1 设计约定

1. **RESTful风格**: URL使用名词复数，HTTP方法语义化（GET查询、POST创建、PUT更新、DELETE删除）
2. **版本控制**: URL路径中携带版本号 `/api/v1/`
3. **分页规范**: 列表接口统一支持 `?page=1&size=20`，响应含 `total`、`page`、`size` 字段
4. **时间格式**: 统一使用ISO 8601格式 `yyyy-MM-ddTHH:mm:ssZ`（UTC时区）
5. **数据脱敏**: 涉及学生个人敏感信息（如手机号）返回时做脱敏处理
6. **行政班确定**: 本学院不设班级，行政班由"年级 + 专业"逻辑确定。所有涉及班级概念的接口均通过 grade 和 major 参数筛选，不依赖班级ID

### 16.2 算分核心逻辑说明

> 本系统仅计算加分部分，不涉及学业成绩。

#### 16.2.1 学术创新成果算分规则

```
总分 = Σ(各子类别分数)，上限7分
子类别内：
  - scoringRule=MAX：取该子类中已通过审核的所有申报项的最高分
  - scoringRule=SUM：累加该子类中所有已通过审核的申报项分数
团队项目（含括号分数的竞赛）：
  - 负责人：取括号外分数
  - 前50%成员：取括号内第一个分数
  - 后50%成员：取括号内第二个分数
```

#### 16.2.2 学生工作算分规则

```
岗位任职（上限3分）：取所有岗位中 (上限分 × 评价权重) 的最大值
学生活动总分（上限5分）：
  - 文体比赛（上限2分）：累加
  - 参与学院大型活动（上限1.2分）：累加
  - 志愿服务和实践（上限1.5分）：累加
  - 党团班活动（上限2分）：
    - 班级活动：出勤 > 总次数×2/3 → +0.5分，不达标不加分
    - 党员活动：出勤 < 总次数×1/2 → -0.5分（扣分项）
    - 团员活动：出勤 < 总次数×1/2 → -0.5分（扣分项）
    - 文明宿舍：一等+1分、二等+0.7分、三等+0.5分
    - 京师智能讲堂等活动：每次+0.1分，上限1分
    - 以上各项累加，总分上限2分
学生工作总分 = 岗位任职得分 + 学生活动总分，上限7分
```

### 16.4 安全考虑

- 所有API请求（除登录）需携带JWT Token
- 接口权限按角色严格校验（学生只能查看/修改本人数据，班委只能操作管辖行政班数据）
- 文件上传需校验文件类型和大小（仅允许jpg/png/pdf，最大10MB；xlsx文件最大20MB）
- 敏感数据在公示前仅对班委和学院管理员可见
- 数据本地化处理，敏感信息不上传云端

### 16.5 性能考虑

- 批量核算接口采用异步任务模式，对大行政班（60+人）返回任务ID，轮询获取结果
- 学院统一加分xlsx导入采用异步处理，支持最多500行
- 统计数据按学年缓存，规则变更或核算触发时清除缓存
