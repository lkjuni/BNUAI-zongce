# 2026-07-12 上午修改记录

本文记录 2026 年 7 月 12 日上午围绕规则树建模、规则配置界面、删除能力、数据库文档和项目说明完成的修改，作为本次 Git 提交的变更说明和后续部署依据。

## 1. 修改概览

本次修改主要解决以下问题：

1. 原规则树将 `total`、`module`、`category`、`subcategory` 固定为不同节点类型，层级扩展不够自然。
2. 规则集、版本和学年配置混在同一页面，节点创建与配置入口命名不清晰。
3. 新建规则节点后缺少可见的删除入口。
4. 学年只有归档逻辑，缺少最高管理员彻底删除全部学年数据的能力。
5. 项目 README、目录结构、迁移说明与当前实现不一致。
6. 核心业务代码缺少解释设计原因和安全边界的注释。

本次提交将规则树收敛为通用树模型，补齐规则节点和学年删除流程，重排规则配置界面，并完善项目文档、迁移脚本和回归验证。

## 2. 规则树模型调整

### 2.1 节点类型合并

规则节点类型由原来的：

```text
total / module / category / subcategory / item
```

调整为：

```text
aggregate / item
```

- `aggregate`：汇总节点，用于总分、模块、分类、子分类等任意层级。
- `item`：具体规则项，用于申报、审核和单项计分。

模块、分类和子分类不再是固定数据库类型，而是由节点名称和 `parent_id` 父子关系自然表达。`parent_id IS NULL` 的节点是该规则版本的根节点。

### 2.2 新增规则树约束

后端新增以下校验：

- 每个规则版本只能有一个根节点。
- 根节点必须是 `aggregate`。
- 父节点必须属于同一规则版本。
- 父节点必须是 `aggregate`，`item` 不能包含子节点。
- 节点不能以自己或自己的下级作为父节点，避免形成循环。
- `aggregate` 不能作为学生申报入口。
- 有子节点的汇总节点不能直接改成 `item`。

数据库新增检查约束：

```sql
CHECK (node_type IN ('aggregate', 'item'))
CHECK (node_type = 'item' OR is_apply_entry = 0)
```

为了兼容旧接口，后端仍可接收 `total/module/category/subcategory`，但写入数据库前统一转换为 `aggregate`。

### 2.3 汇总分与规则项得分

- `max_score` 和 `aggregation_type` 属于汇总节点，用于求和、取最高、封顶等汇总规则。
- 规则项自身如何得分，由“节点配置 → 计分配置”中的固定分、等级分、权重或受控公式决定。
- 前端选择 `item` 时会隐藏并禁用“汇总上限分”和“汇总方式”。

### 2.4 默认规则集同步

`src/defaultRuleSet.js` 和开发演示规则均已改为使用 `aggregate/item`。默认规则集仍保持原有业务层级和计分配置，只改变节点类型表达方式。

## 3. 数据库迁移

新增一次性迁移文件：

```text
sql/migrate-rule-node-aggregate.sql
```

迁移内容：

- 将 `total/module/category/subcategory` 更新为 `aggregate`。
- 修改 `rule_node.node_type` 字段说明。
- 增加 `aggregate/item` 和申报入口检查约束。

本机开发数据库已经完成该迁移。迁移前还修复了一个历史测试版本中的孤立 `item` 根节点，将其重新挂到原有汇总根节点下；没有删除申报、成绩或规则配置数据。

目标环境升级时需要单独执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts\migrate-db.ps1 -MigrationFile migrate-rule-node-aggregate.sql
```

迁移文件只能执行一次。全新数据库直接执行 `sql/schema.sql`，不需要再运行历史迁移。

## 4. 规则配置前端调整

### 4.1 页面结构

规则配置区页签调整为：

```text
规则集与版本 / 创建节点 / 节点配置 / 学年管理
```

具体变化：

- “节点”改名为“创建节点”。
- “配置”改名为“节点配置”。
- 新建规则集和新建版本从“学年”页移到独立的“规则集与版本”页。
- “学年”改名为“学年管理”，只保留学年创建、快照绑定和删除操作。
- 规则树中的规则项显示“按计分配置”，汇总节点显示汇总上限。
- 用户角色增加 `super_admin`（最高管理员）。

### 4.2 节点类型联动

创建汇总节点时显示：

- 汇总上限分
- 汇总方式

创建规则项时显示：

- 是否作为学生申报入口
- 具体分值在“节点配置 → 计分配置”中维护

## 5. 节点删除功能

“节点配置”页新增“删除选中节点”按钮。

删除行为：

- 支持删除选中节点及其全部下级节点。
- 通过递归查询按“子节点优先”顺序删除，满足自引用外键约束。
- 如果节点或下级节点已被申报、单项得分或节点汇总结果引用，返回 `409` 并拒绝删除。
- 有历史业务引用时，应新建规则版本后再调整，不能破坏已有成绩解释链路。
- 删除节点后，如果当前规则版本正被学年使用且已有审核通过的申报，自动触发相关学年重算。
- 删除操作写入 `system_operation_log`。

## 6. 学年删除功能

学年删除分为两个等级。

### 6.1 普通删除或归档

- 学年没有申报、核算和公示数据时，物理删除学年和快照。
- 学年已有业务数据时，仅将状态改为 `archived`。
- 该操作用于日常管理，避免误删历史数据。

### 6.2 最高管理员彻底删除

新增最高管理员彻底删除入口，执行条件：

- 请求提供最高管理员用户 ID。
- 后端查询 `system_user`，确认用户状态为 `active` 且角色为 `super_admin`。
- 操作者必须再次输入完整学年名称确认。

彻底删除在一个数据库事务中按依赖顺序清除：

1. 异议及处理记录
2. 公示批次和公示结果
3. 分数变化、核算任务、错误、警告及各级得分
4. 申报、字段、成员、审核记录、提交版本和材料元数据
5. 审核任务和批量审核记录
6. 学年规则快照
7. 学年记录

数据库事务提交后，再清理 `uploads/applications/<application_id>` 对应的本地材料目录。规则集和规则版本不会随学年删除。

该操作写入 `force_delete_academic_year` 操作日志。

## 11. 已知限制

1. 当前系统仍是业务验证原型，没有正式登录会话。彻底删除学年暂时通过 `X-Operator-Id` 查验最高管理员，正式系统应从可信登录会话取得身份。
2. 历史迁移脚本不是可重复执行脚本，目标数据库执行前必须备份并确认迁移状态。
3. `xlsxLite.js` 只支持本项目需要的简单表格，不是完整 Excel 兼容实现。
4. 规则版本发布后的完全只读约束尚未在所有配置接口统一实施，后续应补充版本复制和发布锁定机制。
