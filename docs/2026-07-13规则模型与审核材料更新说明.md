# 2026-07-13 规则模型与审核材料更新说明

## 一、更新背景

本次更新针对规则配置中“同一规则既可以作为汇总节点，也可以作为申报项”的歧义进行收紧。系统现在明确区分两类节点：

- `aggregate`：只负责汇总子节点得分，可以设置汇总方式和上限分，不能直接接收学生申报。
- `item`：代表一个可以被学生申报的最小规则项，只负责申报字段、证明材料、审核要求和基础计分，不能设置汇总方式或上限分。

规则项的业务约束是：同一学生在同一学年内，每个规则项最多提交一条申报。规则项不是“国家级、省级、校级”等每一个分档，而是一个独立的申报主题；分档应配置在该规则项的表单字段和计分配置中。

## 二、主要修改

### 1. 审核通过规则

- 不采用班委、学院两级串行审核。
- 学委或学院管理员等有权限的审核角色，任意一个对申报执行“通过”，申报立即变为 `approved`。
- 最终通过后立即触发自动核算；前端会显示“申报已通过”和“已自动触发核算”。
- `need_second_audit` 作为历史字段保留，但不再参与审核状态分流。

### 2. 规则树模型

- 删除旧的 `module`、`category`、`subcategory` 节点类型，统一为 `aggregate` 和 `item`。
- 规则项必须是申报入口，且必须挂在汇总节点下面。
- 汇总节点不能作为申报入口。
- 规则项不能拥有子节点。
- 汇总节点不能只有一个子节点，以避免没有业务含义的冗余层级。
- 计分配置、申报字段、材料要求、审核要求、团体分配规则只能配置在规则项上。
- 发布规则版本时，后端会统一检查以上约束；不符合约束时拒绝发布。

### 3. 规则项唯一申报

`application_record` 增加 `rule_item_code`，并建立以下唯一约束：

```text
(academic_year_id, student_id, rule_item_code)
```

该约束由数据库和后端同时保证。已撤回、已驳回的旧申报仍然占用该规则项的名额，避免通过重新提交绕过“每项最多一条”的业务规则；用户可以编辑原申报并重新提交。

### 4. 默认规则集

默认规则集已经按照上述模型重新组织。可以并存的成果拆成不同规则项，例如论文、程序设计竞赛、创意策划竞赛、教师素养、文体比赛、实践活动和认证等；同一成果内部的国家级、省级、奖项等级、作者排名等差异，通过规则项的表单字段和计分公式表达。

默认规则集加载后包含：

- 15 个汇总节点；
- 34 个规则项；
- 34 个申报入口；
- 每个规则项均有表单、材料、审核和计分配置；
- 不包含只有一个子节点的冗余汇总节点。

### 5. 审核材料预览

审核详情页现在直接展示学生上传的证明材料：

- PDF 在页面内嵌预览；
- 图片直接展示缩略图；
- 其他文件显示文件名、类型和新窗口打开链接；
- 同时显示材料审核状态，方便班委核对材料后执行通过、驳回或退回修改。

### 6. 自动核算保持不变并得到规则约束保护

核算仍然由业务事件自动触发：

- 某条申报审核通过后，自动重新计算对应学生的成绩；
- 规则快照发生变化且学年中存在已通过申报时，自动重新计算；
- 先计算每个规则项的基础分，再沿规则树执行求和、取最高、封顶等汇总规则。

本次更新补充了论文按共同第一作者人数折算、实践活动奖励加分、直接使用审核字段计分、数量乘单价并封顶等公式模板。

## 三、数据库与初始化

全量建表脚本是 [sql/schema.sql](../sql/schema.sql)。规则项唯一申报的历史迁移脚本是 [sql/005_unique_rule_item_application.sql](../sql/005_unique_rule_item_application.sql)。

需要干净测试环境时执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\init-db.ps1
```

该脚本会清空并重建数据库表。重建完成后默认只初始化演示账号和演示学生，不自动加载规则集；登录管理员后点击“载入默认规则集”即可开始测试。
