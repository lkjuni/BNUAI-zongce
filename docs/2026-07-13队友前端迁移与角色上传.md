# 队友前端迁移与角色上传说明

## 1. 本次目标

本次工作以队友提供的学生端、学委端和管理员端页面为外观与角色入口参考，将项目当前已经实现的业务机制完整迁入统一前端。迁移时不再使用旧 `/api/v1` 接口和页面内模拟数据，统一调用项目现有 `/api` 接口。

本次新增范围只包括：

- 真实角色登录和登录会话；
- 学委统一上传；
- 学院统一上传。

原有规则树、学年快照、申报材料、审核、自动核算、结果、公示、统计、导出、学生管理、用户管理和操作日志均保留。

## 2. 前端迁移结果

页面沿用队友方案中的深蓝顶栏、深色侧栏、青色主操作色、紧凑表格和工作台布局，并增加登录入口。登录后按角色显示菜单：

| 角色 | 工作台功能 |
|---|---|
| 学生 | 申报、材料提交、本人申报记录、结果 |
| 学委 | 审核、学委统一上传、结果 |
| 学院管理员 | 全部规则、学年、申报、审核、上传、核算、结果和系统管理功能 |
| 最高管理员 | 学院管理员全部功能，以及彻底删除学年 |

管理员规则配置仍使用当前通用规则树模型：节点只区分 `aggregate` 与 `item`，任意层级由 `parent_id` 表达。规则集删除、节点删除和学年删除入口均保留。

## 3. 统一上传机制

统一上传的目标是让不适合学生逐条申报的学院统一数据进入同一套核算链路，而不是提供一个绕过规则的“改总分”入口。

处理步骤如下：

1. 上传人选择学年和当前规则快照中的目标规则项。
2. 系统读取 xlsx 中的学号、项目名称、分数和说明。
3. 学委上传时，逐行校验学生是否属于学委账号关联的行政班。
4. 每个成功行生成一条状态为 `approved` 的受信申报及提交/审核记录。
5. `score_import_batch` 保存整批汇总，`score_import_row` 保存每一行成功或失败原因。
6. 全部行处理完成后只触发一次自动核算。
7. 核算先将 `import_score` 写入规则项得分，再执行取最高、求和、封顶等规则树汇总。

学院上传与学委上传的差异只有权限范围：学委限本行政班，学院管理员和最高管理员可覆盖全院。

## 4. 数据库与接口

新增数据表：

- `auth_session`：登录令牌摘要、有效期和撤销状态；
- `score_import_batch`：统一上传批次；
- `score_import_row`：上传逐行处理结果。

新增接口：

```text
POST /api/auth/login
GET  /api/auth/me
POST /api/auth/logout
PUT  /api/auth/password

GET  /api/imports/template?scope=committee|college
GET  /api/imports/entries?academicYearId=学年ID
POST /api/imports/committee
POST /api/imports/college
GET  /api/imports/history
```

迁移文件为 `sql/004_auth_role_score_import.sql`。

## 5. 验证结果

- 学生登录后仅显示“申报、结果”，并自动使用账号关联学生 ID。
- 学委登录后仅显示“审核、统一上传、结果”。
- 学院管理员登录后保留全部 10 个现有模块。
- 学委调用学院上传接口返回 `403`。
- 学院上传成功后自动生成核算批次，规则项结果来源为 `import`。
- 学委上传成功后自动生成核算批次，上传历史可查询。
- 当前默认学年可在上传页面加载 13 个目标规则项。
- 桌面端和 390×844 窄屏均已检查，无页面级横向内容溢出。
