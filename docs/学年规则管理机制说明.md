# 学年、规则、快照机制说明

这份文档面向业务人员和普通用户，尽量不使用数据库术语。它解释系统里为什么会有“学年”“规则集”“规则版本”“规则快照”，以及它们和学生申报、审核、核算之间是什么关系。

## 1. 先用一句话说明

系统不是直接拿“正在编辑的规则”给学生填报和算分，而是：

1. 管理员先配置一套规则。
2. 把这套规则发布成一个版本。
3. 给某个学年绑定这个版本。
4. 绑定时系统复制一份当时的规则，形成“学年规则快照”。
5. 学生申报、班委审核、系统核算、公示结果，都以这个学年的规则快照为准。

这样做的核心目的是：避免管理员修改规则时，把已经开始申报或已经审核过的数据弄乱。

## 2. 什么是学年

学年就是一次综测业务周期，例如：

- 2024-2025 学年
- 2025-2026 学年

一个学年通常包含这些阶段：

- 规则配置
- 学生申报
- 班委审核
- 学院审核
- 自动核算
- 结果公示
- 复查/异议
- 归档

所以“学年”不是单纯的年份文字，而是一次完整综测工作的容器。

## 3. 什么是规则集

规则集可以理解为“综测细则模板”。

例如：

```text
人工智能学院本科生综合测评规则集
```

它里面包含很多规则项，比如科研竞赛、志愿服务、学生工作、文体活动等。管理员可以预设多个规则集，就像快递系统里可以预设多个寄件人信息簿。

规则集本身是一个模板，不直接参与学生申报和算分。

## 4. 什么是规则版本

规则每年可能调整，所以同一个规则集下面会有多个版本。

例如：

```text
人工智能学院本科生综合测评规则集
├─ 2024-2025 版
└─ 2025-2026 版
```

规则版本解决的问题是：同一个学院的规则会演进，但历史规则不能丢。

例如 2025-2026 学年可能发生这些变化：

- 新增某个竞赛项目
- 修改某类奖项分值
- 调整某类项目最高加分
- 修改团队成果分配比例
- 增加或减少证明材料要求

这些变化应该形成一个新版本，而不是直接覆盖旧规则。

## 5. 什么是规则快照

规则快照就是“某个学年正式采用的规则副本”。

例如管理员决定：

```text
2025-2026 学年
使用
人工智能学院本科生综合测评规则集 v2025-2026
```

系统会在绑定时复制一份规则，形成：

```text
2025-2026 学年规则快照
```

从此以后，这个学年的学生填报界面、审核要求、算分逻辑、公示结果，都看这份快照。

## 6. 为什么不能直接使用正在编辑的规则

如果学生申报界面直接依赖管理员正在编辑的规则，会出现严重问题。

例如：

1. 学生 A 已经按旧规则提交了“国家级竞赛一等奖”。
2. 班委已经审核通过。
3. 管理员后来把这个规则项改名、改分值或删除。
4. 那么学生 A 的申报到底还算不算？按几分算？审核记录是否还有效？

这会导致：

- 前面填报的数据可能失效
- 审核标准前后不一致
- 已审核记录无法解释
- 核算结果不稳定
- 公示时难以追责

所以系统必须区分：

- 正在编辑的规则模板
- 某学年已经确定采用的规则快照

## 7. 规则采用树形结构建模

综测规则天然是树状结构。

例如：

```text
规则集
├─ 总分
│  ├─ 思想品德
│  ├─ 学业成绩
│  ├─ 学术创新成果
│  │  ├─ 科研成果
│  │  │  ├─ 论文成果
│  │  │  └─ 项目成果
│  │  └─ 竞赛获奖
│  │     ├─ 创新创业类竞赛
│  │     └─ 学科竞赛
│  └─ 学生工作
│     ├─ 岗位任职
│     └─ 文体活动
└─ 团体分配规则
```

树上的每个点都叫“规则节点”。

## 8. 规则节点有哪些类型

系统只区分两类节点：

```text
aggregate    汇总节点
item         具体规则项
```

过去所说的总分、模块、分类、子分类，本质上都是 `aggregate`。系统根据 `parent_id` 形成树，不再要求规则必须正好分成固定的四层。

```text
综合测评总分（根汇总节点，parent_id 为空）
├─ 学术创新成果（汇总节点）
│  └─ 科研成果（汇总节点）
│     └─ 论文成果（汇总节点）
│        └─ 论文成果认定（规则项）
└─ 学生工作（汇总节点）
```

因此，将来出现更多或更少层级，只需改变父子关系，不需要增加节点类型或修改程序。`item` 是否出现在学生填报界面，另外由“是否作为申报入口”控制。

例如：

```text
论文成果等级认定
挑战杯/互联网+竞赛
文体比赛获奖
班级与团支部职务
```

## 9. 每个规则节点需要配置什么

两类节点承担的职责不同。

### 汇总节点通常配置

- 节点名称
- 所属上级节点
- 分值上限
- 类内规则
- 汇总方式

汇总节点不能直接作为学生申报入口。每个规则版本只能有一个根汇总节点。

例如：

```text
科研竞赛类
最高加 8 分
同一作品只取最高奖项
```

### 规则项节点通常配置

- 规则项名称
- 申报字段
- 证明材料要求
- 审核要求
- 算分方式
- 是否允许重复申报
- 适用对象

例如：

```text
国家级竞赛一等奖
学生需要填写：竞赛名称、获奖等级、本人角色
学生需要上传：获奖证书、团队成员证明
审核人需要核对：奖项等级、成员排名、证书真实性
算分：基础分 5 分，再按团队角色比例折算
```

## 10. 学生填报界面从哪里来

学生填报界面不是程序员为每个规则项单独写死的。

它来自规则快照里的“申报字段”和“材料要求”。

例如某个规则项配置了：

```text
字段：
- 论文题目
- 论文等级
- 作者顺序

材料：
- 论文检索页
- 录用证明
```

那么学生看到的填报界面就会自动包含这些内容。

如果另一个规则项配置了：

```text
字段：
- 比赛名称
- 获奖等级
- 本人角色

材料：
- 获奖证书
- 团队成员证明
```

学生看到的界面也会相应变化。

## 11. 规则发生变化怎么办

规则变化分两种情况。

### 情况一：还没有开始申报

管理员可以继续修改规则版本，然后重新绑定学年快照。

这时影响不大，因为还没有学生数据。

### 情况二：已经有人申报或审核

这时不能随意修改正在使用的规则快照。

推荐做法是：

- 小改动：通过受控配置修改，并触发该学年重新核算
- 大改动：新建规则版本，经确认后重新绑定学年快照
- 已提交申报：保留原始申报记录和提交版本
- 已审核记录：保留审核过程，便于追溯

也就是说，系统允许规则调整，但必须留痕，并且要能重新核算。

## 12. 核算什么时候发生

核算不是学生点击提交时发生。

核算通常由这些事件触发：

- 某条申报审核通过
- 批量审核通过
- 学年规则快照发生变化
- 管理员手动重算

原因是：学生提交只是“我申报了这件事”，还没有经过审核确认。只有审核通过后，这条申报才应该进入得分计算。

## 13. 核算分两步

第一步：计算每条已审核通过申报的基础得分。

例如：

```text
国家级竞赛一等奖：5 分
省级竞赛二等奖：2 分
志愿服务 40 小时：2 分
```

第二步：按规则树向上汇总。

例如：

```text
科研竞赛类：
原始分 = 5 + 2 = 7
类内上限 = 6
最终计入 = 6
```

这样既能解释单条申报得分，也能解释分类上限、取最高、封顶等规则。

## 14. 公示为什么也要绑定核算批次

公示不是直接展示“实时计算结果”。

公示应该展示某一次确定的核算结果。

例如：

```text
2025-2026 学年
第 1 次正式核算结果
发起公示
```

公示开始后，即使后台又发生重新核算，本次公示也应该能追溯当时展示的结果。

所以公示需要绑定一个核算批次，并把公示结果固化下来。

## 15. 一条完整业务链路

```text
管理员配置规则集
  ↓
发布规则版本
  ↓
给学年绑定规则快照
  ↓
学生按快照填报并上传材料
  ↓
班委/学院按快照审核
  ↓
审核通过触发自动核算
  ↓
系统生成核算批次和结果
  ↓
班委或学院管理员发起公示
  ↓
学生查看公示结果并提出异议
  ↓
处理异议后归档
```

## 16. 给用户的最简解释

可以这样对用户解释：

> 规则集是可维护的规则模板；规则版本是每次规则调整后的正式版本；学年是一次综测周期；规则快照是某个学年真正采用的规则副本。学生申报、审核、算分和公示都基于学年快照，而不是直接基于正在编辑的规则模板。这样可以保证规则调整时，已经提交和审核的数据不会混乱，同时又能支持重新核算和历史追溯。
